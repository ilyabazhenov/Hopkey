# Выпуск релиза Hopkey

Как собрать и опубликовать новую версию Hopkey. Весь процесс автоматизирован
скриптом [`release.sh`](release.sh) — он прогоняет тесты, собирает приложение и
**два** артефакта, **сам коммитит и пушит** релизный коммит с тегом, затем создаёт
GitHub Release. Каждый релиз содержит:

- **`Hopkey-<ver>.zip`** — для **автообновления** через Sparkle. На него ссылается
  `appcast.xml`; уже установленный Hopkey скачивает и ставит его сам.
- **`Hopkey-<ver>.dmg`** — для **первой ручной установки** (drag-to-Applications).

Подпись — стабильная self-signed («Hopkey Dev»), **без нотаризации Apple**, поэтому
в release notes автоматически добавляется памятка про обход Gatekeeper (`xattr`).

---

## Контракт агента

Без явного подтверждения пользователя агент:

- не выбирает номер версии сам — берёт из явного указания пользователя;
- всегда включает в release notes **список изменений относительно предыдущей
  версии**. `release.sh` собирает его автоматически из git-истории (коммиты между
  прошлым тегом и HEAD) и **падает, если список пуст**. Перед публикацией агент
  просматривает список и при необходимости курирует его — передав готовый текст
  через `CHANGELOG`:

  ```bash
  CHANGELOG=$'- Universal binary: запуск на Apple Silicon и Intel\n- Фикс локализации' ./release.sh
  ```

Остальное `release.sh` обеспечивает сам и падает, если условие не выполнено:

- **`swift test`** прогоняется перед сборкой — красные тесты останавливают релиз;
- **чистота дерева** проверяется до сборки: допускается только правка `VERSION`
  (она и есть бамп), любые другие незакоммиченные изменения — стоп. Так релизный
  коммит не вберёт постороннего;
- **коммит, тег и push идут ДО создания GitHub Release** — поэтому тег всегда
  указывает на релизный коммит (а не на устаревший HEAD remote). Отдельный ручной
  шаг «закоммитить фид» больше не нужен.

## Платформа и архитектура

С **v1.3.0** релиз — **universal binary** (`arm64 + x86_64`): `build.sh` собирает обе арки
(`swift build --arch arm64 --arch x86_64`, флаг `UNIVERSAL=1` по умолчанию) и проверяет
наличие обоих слайсов через `lipo`. Sparkle.framework уже universal. Один артефакт запускается
и на Apple Silicon, и на Intel.

| Сценарий | arm64 Mac | Intel Mac |
|----------|-----------|-----------|
| Ручная установка из DMG | ✅ | ✅ |
| Автообновление Sparkle | ✅ | ✅ |

> Dev-сборки (`make run`/`watch`) идут под текущую арку (`UNIVERSAL=0`) ради скорости —
> релиза это не касается. На Apple Silicon x86_64-слайс «вживую» не проверить (Rosetta берёт
> arm64) — доверяем `lipo`.

## Предусловия (проверить один раз)

```bash
gh auth status          # залогинен (аккаунт ilyabazhenov)
git remote -v           # origin → github.com/ilyabazhenov/Hopkey
xcrun --find hdiutil    # встроен в macOS, ставить отдельно не нужно
```

Разовая настройка автообновления (если ещё не сделана):

- `./setup-signing.sh` — стабильный серт «Hopkey Dev» (иначе обновление
  подпишется ad-hoc и Sparkle отвергнет его из-за несовпадения идентичности);
- `…/Sparkle/bin/generate_keys` — EdDSA-ключ (приватный в Keychain; публичный уже
  прописан в `build.sh` → `SUPublicEDKey`). **Сделайте бэкап приватного ключа**
  (`generate_keys -x файл`) — без него нельзя выпускать обновления, которые примет
  установленная база.

> ⚠️ **Доступ к фиду.** Sparkle качает `appcast.xml` и zip **анонимно**. Если
> репозиторий приватный, и фид, и ассеты отдают 404 — автообновление не работает.
> Для рабочего автообновления репозиторий (или отдельный репо с релизами) должен
> быть публичным.

## Версионирование

- Схема — [SemVer](https://semver.org): `MAJOR.MINOR.PATCH`, тег — `v1.2.0`.
- **Единственный источник правды** — файл `VERSION` в корне (одна строка). Его
  читает `build.sh` и проставляет в `Info.plist`; тег и appcast берут то же число.
- Поднять версию = изменить `VERSION`; коммитит его уже `release.sh` (в составе
  релизного коммита вместе с `appcast.xml`).

## Шаги релиза

### 0. Предполётная проверка

```bash
git switch main && git pull --ff-only
```

Рабочее дерево должно быть чистым (кроме `VERSION`, см. ниже). `swift test` и
проверку чистоты дерева `release.sh` прогоняет сам в начале и падает при провале.

### 1. Поднять версию

```bash
echo "1.2.0" > VERSION
```

Коммитить вручную не нужно — `release.sh` закоммитит `VERSION` вместе с
`appcast.xml` одним релизным коммитом.

### 2. Запустить релиз

```bash
./release.sh
```

Скрипт сам выполнит по порядку: предполётная проверка (чистое дерево + `swift
test`) → `build.sh` → `Hopkey-1.2.0.zip` + подпись EdDSA и генерация `appcast.xml`
→ **коммит `release v1.2.0` (`VERSION` + `appcast.xml`) + тег `v1.2.0` + push в
`main`** → `Hopkey-1.2.0.dmg` (с ярлыком `/Applications`) → создание GitHub Release
`v1.2.0` из уже запушенного тега (`--verify-tag`) с обоими файлами и готовыми
release notes (список изменений относительно прошлого тега + памятка про `xattr`).

Список изменений собирается автоматически из git-истории. Чтобы задать
курированный текст вместо авто-списка:

```bash
CHANGELOG=$'- Первый пункт\n- Второй пункт' ./release.sh
```

Если список изменений собрать не удалось (пусто), `release.sh` остановится — это
защита от публикации релиза без описания.

> DMG собирается **после** appcast специально — чтобы образ не попал в фид
> обновлений. В `appcast.xml` всегда ровно одна запись на актуальный zip.

> **Почему коммит и тег создаются ДО релиза.** `appcast.xml` рождается во время
> сборки, поэтому релизный коммит можно создать только в середине процесса. Если
> сначала вызвать `gh release create`, тег встанет на текущий HEAD remote-ветки —
> устаревший, ведь `appcast.xml`/`VERSION` ещё не запушены. Так теги `v1.4.0` и
> `v1.5.0` изначально указывали не на тот коммит. Поэтому `release.sh` сперва
> пушит релизный коммит и тег, а релиз создаёт с `--verify-tag`.

### 3. Проверка

```bash
gh release view v1.2.0 --web
git ls-remote --tags origin v1.2.0   # тег указывает на коммит «release v1.2.0»
```

Финально (на «чистой» машине/пути): автообновление — установить старую версию,
нажать «Проверить обновления…», убедиться что Hopkey обновляется и
перезапускается. Ручная установка — смонтировать DMG, перетащить в Applications,
снять карантин, запустить.

## Почему `xattr`, а не «правый клик → Открыть»

Self-signed-подпись для Gatekeeper эквивалентна отсутствию валидной. Для
скачанного из интернета бандла macOS нередко показывает не «неизвестный
разработчик» (чинится правым кликом), а «приложение повреждено» — этот диалог
правым кликом **не** обходится. Надёжный способ — снять атрибут карантина:
`xattr -dr com.apple.quarantine /Applications/Hopkey.app`. Поэтому в памятке
даём именно его.

> Если появится платный Apple Developer ID — стоит перейти на Developer ID +
> нотаризацию (`notarytool` + `stapler`); тогда DMG откроется двойным кликом без
> терминала, а Sparkle сможет проверять обновления и по нотаризации.
