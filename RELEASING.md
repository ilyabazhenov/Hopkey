# Выпуск релиза Hopkey на GitHub

Инструкция для агента (и человека): как собрать, упаковать и опубликовать
новую версию Hopkey как GitHub Release. Hopkey распространяется как `.dmg`
с приложением внутри. Подпись — **ad-hoc** (платного Apple Developer ID нет),
поэтому в релиз обязательно входит памятка про обход Gatekeeper.

---

## Контракт агента

Чего агент **не делает** без явного подтверждения пользователя:

- не выбирает номер версии сам — спрашивает или берёт из явного указания
  пользователя;
- не пушит теги и не создаёт релиз, пока сборка и тесты не прошли локально;
- не публикует релиз, если рабочее дерево грязное (есть незакоммиченные
  изменения) — сначала коммит/стэш.

Что агент делает сам: сборку, упаковку DMG, генерацию черновика release notes
из git-лога, проверки из чек-листа.

---

## Предусловия (проверить один раз)

```bash
gh auth status          # должен быть залогинен (аккаунт ilyabazhenov)
git remote -v           # origin → github.com/ilyabazhenov/Hopkey
xcrun --find hdiutil     # hdiutil встроен в macOS, отдельно ставить не нужно
```

## Версионирование

- Схема — [SemVer](https://semver.org): `MAJOR.MINOR.PATCH` (например `1.2.0`).
- Тег в git — с префиксом `v`: `v1.2.0`.
- **Единственный источник правды для номера** — файл `VERSION` в корне
  репозитория (одна строка, например `1.2.0`). Его читает `build.sh` и
  проставляет в `Info.plist`; git-тег ставится из того же числа.
- Поднять версию = изменить `VERSION` и закоммитить. Этот коммит и есть
  человекочитаемая фиксация «мы выпускаем 1.2.0».
- Env-переменная `VERSION=… ./build.sh` переопределяет файл — только для
  пробных/локальных сборок, в релиз не идёт.

Дальше по тексту обозначаем выбранную версию как `1.2.0` — подставляйте свою.

## Шаги релиза

### 0. Предполётная проверка

```bash
git switch main && git pull --ff-only
git status --porcelain        # должно быть пусто
swift test                    # тесты ядра обязаны пройти
```

Если рабочее дерево не пустое — остановиться и разобраться с изменениями
(закоммитить или отложить), затем повторить.

### 1. Поднять версию

Записать новый номер в файл `VERSION` и закоммитить:

```bash
echo "1.2.0" > VERSION
git commit -am "chore: bump version to 1.2.0"
git push
```

### 2. Сборка release-бандла

```bash
./build.sh        # версия подхватится из файла VERSION
```

На выходе — `build/Hopkey.app` с проставленными
`CFBundleShortVersionString`/`CFBundleVersion = 1.2.0`. Проверить:

```bash
/usr/libexec/PlistBuddy -c 'Print :CFBundleShortVersionString' \
    build/Hopkey.app/Contents/Info.plist     # → 1.2.0
```

### 3. Упаковка в DMG

```bash
VERSION=1.2.0
DMG="build/Hopkey-${VERSION}.dmg"
STAGING="build/dmg"

rm -rf "$STAGING" "$DMG"
mkdir -p "$STAGING"
cp -R "build/Hopkey.app" "$STAGING/"
ln -s /Applications "$STAGING/Applications"     # drag-to-install
hdiutil create -volname "Hopkey ${VERSION}" \
    -srcfolder "$STAGING" -ov -format UDZO "$DMG"
rm -rf "$STAGING"

ls -lh "$DMG"     # убедиться, что файл создан
```

Получится `build/Hopkey-1.2.0.dmg` с приложением и ярлыком `/Applications` —
пользователь монтирует и перетаскивает иконку в папку приложений.

### 4. Тег

```bash
git tag -a v1.2.0 -m "Hopkey v1.2.0"
git push origin v1.2.0
```

### 5. Release notes

Сгенерировать черновик из коммитов с прошлого тега (для самого первого релиза
`--generate-notes` возьмёт всю историю):

```bash
git log $(git describe --tags --abbrev=0 HEAD^ 2>/dev/null)..HEAD \
    --pretty='- %s' 2>/dev/null || git log --pretty='- %s'
```

Привести к человеческому виду: сгруппировать по «Новое / Исправления / Прочее»,
выкинуть служебные коммиты. Показать пользователю на утверждение перед публикацией.

### 6. Создание GitHub Release

```bash
gh release create v1.2.0 \
    "build/Hopkey-1.2.0.dmg" \
    --title "Hopkey 1.2.0" \
    --notes-file -    # текст подаётся со stdin; см. шаблон ниже
```

В тело релиза **обязательно** включить памятку про Gatekeeper (ниже), иначе
пользователи решат, что приложение «сломано».

После публикации — проверить:

```bash
gh release view v1.2.0 --web
```

## Шаблон тела релиза

```markdown
## Hopkey 1.2.0

### Новое
- …

### Исправления
- …

---

### Установка

1. Скачайте `Hopkey-1.2.0.dmg` ниже и откройте его.
2. Перетащите **Hopkey** в папку **Applications**.
3. Приложение подписано ad-hoc (не нотаризовано Apple), поэтому при первом
   запуске macOS заблокирует его. Снимите карантин одной командой в Терминале:

   ```bash
   xattr -dr com.apple.quarantine /Applications/Hopkey.app
   ```

   После этого запускайте обычным двойным кликом. Hopkey живёт в строке меню
   и потребует разрешение **Универсальный доступ** (Accessibility) для
   глобального хоткея.
```

---

## Почему именно `xattr`, а не «правый клик → Открыть»

Ad-hoc-подпись для Gatekeeper эквивалентна отсутствию валидной подписи. Для
скачанного из интернета бандла macOS нередко показывает не «неизвестный
разработчик» (который чинится правым кликом), а «приложение повреждено» —
этот диалог правым кликом **не** обходится. Надёжный способ для всех случаев —
снять атрибут карантина через `xattr -dr com.apple.quarantine`. Поэтому в
памятке даём именно его.

> Если в будущем появится платный Apple Developer ID — стоит перейти на
> полноценную подпись Developer ID + нотаризацию (`notarytool`) и `stapler`;
> тогда DMG будет открываться двойным кликом без команд в терминале, и этот
> раздел можно будет убрать.
